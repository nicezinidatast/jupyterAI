## 1. Admin (관리자)

| 항목 | 내용 |
|---|---|
| **역할 코드** | `Admin` |
| **부서 규모 내 인원** | 1~3명 |
| **권한** | 사용자/역할/커넥션/PII 정책/감사 정책 전반 관리. 노트북 분석은 보조적 |
| **기술 수준** | 인프라/보안 운영 친화. SQL 가능, Python은 선택 |
| **플랫폼 사용 빈도(추정)** | 매일 30분~1시간, 주요 사건(권한 변경·신규 커넥션·인시던트)에 따라 변동 |

### 1.1 목표 (Goals)
- 부서의 데이터 접근/PII 통제를 **하나의 콘솔**에서 일원화한다.
- 모든 데이터 액세스가 **감사 가능**하도록 정책을 유지한다.
- 분석가가 분석을 **자율적으로** 수행하면서도 위험 활동은 즉시 식별 가능해야 한다.

### 1.2 페인포인트 (Pain Points — 추정)
- 현재는 DB 접근권한이 도구별/사람별로 분산되어 **권한 회수가 누락**되는 경우가 있다.
- 분석가들이 SQL 클라이언트·로컬 노트북·메신저로 결과를 옮겨 다녀 **감사 추적이 어렵다**.
- PII 마스킹 규칙이 도구마다 달라 **일관성**이 없다.
- 신규 인력 온보딩/이직자 오프보딩이 **수작업**으로 느리다.

### 1.3 일과 시나리오 (Daily Scenario)
1. 아침: 야간 감사 로그·시스템 헬스 대시보드 확인 (5분).
2. 오전: 신규 입사자에게 역할 할당, 부서별 커넥션 권한 부여.
3. 점심 후: 분석가 요청으로 새 DB 커넥션 등록(공용 자격증명 + 권한 화이트리스트).
4. 오후: PII 마스킹 규칙 1건 신규 추가, 영향 받는 컬럼 점검.
5. 저녁: 분기 권한 리뷰 보고서 자동 생성 결과 검토.

## 2. Analyst (분석가 — SQL + Python/R 통합)

> **A1 적용**: 원래 분리되어 있던 Data Scientist를 통합. 권한·도구·UX는 동일. 모델링/통계 작업을 더 깊게 하는 사용자도 같은 역할로 묶임. 필요 시 본 페르소나 안의 sub-flow(예: SQL-Centric vs Notebook-Centric)로 구분.

| 항목 | 내용 |
|---|---|
| **역할 코드** | `Analyst` |
| **부서 규모 내 인원** | 15~50명 (Analyst 10~40 + DS 5~10 통합) |
| **권한** | 부여된 커넥션에 대해 SQL 실행, 셀 단위 Python/R 실행, 파일 업로드, 노트북 작성·공유 |
| **기술 수준** | SQL 일상 사용. 파이썬 활용도는 사용자별 편차 큼(EDA 수준 ~ ML 모델링) |
| **플랫폼 사용 빈도(추정)** | **하루 평균 3~5시간** — 본 플랫폼이 1차 작업 도구 |

### 2.1 목표 (Goals)
- 한 곳에서 **SQL + 파이썬 + 시각화**를 끝낸다. 도구·창 전환 없이.
- 분석 결과를 **노트북 단위로 공유**해 동료 리뷰를 받는다.
- 반복 분석을 **재실행 가능**한 형태로 보존한다 (Git 자동 커밋).
- 외부 도구로 데이터를 옮기지 않아도 인터랙티브 차트로 **즉시 의사결정**에 활용한다.

### 2.2 페인포인트 (추정)
- 사내 DB가 여러 종류(PG/MySQL/Oracle/Hive 등)라 **드라이버/접속 세팅이 매번 다름**.
- 로컬 노트북은 **재현이 안 되고**, 권한도 분산되어 동료에게 결과만 캡처해서 보냄.
- 큰 결과 다운로드 후 엑셀에서 다루다 **데이터 누설/PII 노출 리스크** 자체 점검에 시간 소비.
- LLM 어시스턴트를 쓰고 싶지만 **외부 전송 가능 여부**가 불확실해 사용 망설임.
- 5초 이상 걸리는 쿼리에서 **세션이 끊기거나 UI가 응답 없음** 상태에 빠지는 경험.

### 2.3 일과 시나리오
**SQL-Centric 흐름 (Golden Path A)**
1. SSO로 로그인 → 사이드 패널에서 등록된 PostgreSQL 커넥션 선택.
2. 스키마 트리에서 `sales.orders` 더블클릭 → SQL 에디터에 미리보기 쿼리 자동 삽입.
3. 5~10번 쿼리 반복하며 노트북 셀에 저장. PII가 들어간 컬럼은 자동 마스킹 표시 확인.
4. 결과 셀에서 **차트 변환** 클릭 → Plotly 인터랙티브 차트로 즉시 확인.
5. 노트북 저장 → 백그라운드로 Git 자동 커밋. 동료에게 노트북 링크 공유(읽기 권한).

**Notebook-Centric 흐름 (Golden Path B)**
1. SSO 로그인 → 노트북 신규 → CSV 800MB 업로드(NAS 공유 폴더 경유).
2. Python 셀에서 pandas로 EDA → matplotlib 정적 차트.
3. 모델링 코드 작성, 5초 이상 걸리는 셀은 백그라운드 실행 + 알림 옵션 사용.
4. 결과 그래프 + 모델 메트릭을 노트북에 정리 → Git 커밋 + 팀 워크스페이스에 push.
5. 비즈니스 사용자에게 읽기 전용 링크 공유.

## 3. Viewer (비즈니스 사용자)

| 항목 | 내용 |
|---|---|
| **역할 코드** | `Viewer` |
| **부서 규모 내 인원** | 5~20명 |
| **권한** | 공유된 노트북/대시보드 **읽기 전용**, 결과 다운로드(권한 부여 시), 쿼리/파일 업로드 불가 |
| **기술 수준** | SQL/파이썬 비숙련. 대시보드 클릭·필터·다운로드 정도 |
| **플랫폼 사용 빈도(추정)** | 주 1~3회, 회의 준비/의사결정 시점에 집중 |

### 3.1 목표 (Goals)
- 분석가가 공유한 **노트북/대시보드를 보고 의사결정**에 활용한다.
- 가능하면 **필터를 바꿔가며** 인터랙티브하게 본다.
- 결과를 회의 자료(PDF/링크)로 **자가 배포**한다.

### 3.2 페인포인트 (추정)
- 분석 결과를 받으려고 **분석가에게 매번 요청**해야 했고 응답이 느림.
- 받은 자료가 **시점이 어긋남**(어제 기준 vs 오늘 기준).
- 다운로드 받은 파일을 메신저로 돌리다 **데이터 거버넌스 위반**.

### 3.3 일과 시나리오
1. 회의 30분 전, 메신저로 받은 노트북 링크 클릭 → SSO 로그인.
2. 인터랙티브 차트에서 **부서 필터**를 변경하며 본인 관심 영역 확인.
3. PDF 다운로드(Phase 2) 또는 URL 그대로 공유.
4. 회의에서 화면 공유로 활용.

## 4. Security Auditor (보안 감사관) — **신규 (Q-USR-1=D)**

| 항목 | 내용 |
|---|---|
| **역할 코드** | `Auditor` |
| **부서 규모 내 인원** | 1~2명 (정보보안팀/내부감사팀에서 겸직 가능) |
| **권한** | **감사 로그 조회 전용** + PII 마스킹 정책 read-only + 사용자/커넥션 메타데이터 read-only. 데이터 본문 직접 조회·다운로드 권한 없음 |
| **기술 수준** | 컴플라이언스/보안 정책 친화, SQL 기초, 정규식·로그 분석 가능 |
| **플랫폼 사용 빈도(추정)** | 주 1~2회 정기 점검 + 인시던트 발생 시 임시 |

### 4.1 목표 (Goals)
- 부서의 **데이터 접근 활동 전수**를 추적 가능하도록 보장한다.
- PII/민감 데이터의 **외부 송신 여부** 확인 (LLM Phase 2 포함).
- 분기 권한 리뷰의 **공식 출처**를 본 플랫폼으로 일원화한다.
- 인시던트 발생 시 **사용자·시간·쿼리·결과 크기**를 즉시 조회한다.

### 4.2 페인포인트 (추정)
- 현재는 여러 도구의 로그를 SIEM에 모아도 **사용자 세션·쿼리 연결고리**가 끊긴다.
- PII 마스킹이 **언제 어떤 컬럼에 적용**되었는지 이력 추적이 어렵다.
- 권한 회수 후에도 **세션이 살아있을 가능성**을 확인할 수단이 부족하다.

### 4.3 일과 시나리오
1. 주간 점검: 감사 로그 검색 콘솔에서 "지난 7일간 큰 결과 다운로드 Top 20" 쿼리.
2. PII 정책 변경 이력 검토. 변경 적용 후 영향 범위 보고.
3. 인시던트: 특정 사용자 ID로 모든 활동(로그인·쿼리·파일·권한 변경) 타임라인 조회.
4. 분기 리뷰: 권한-사용 매트릭스 자동 생성 결과를 SIEM/관리부서에 송부.
